[abstract]
Chronos is a next-generation systems programming language that unifies features from C++, Rust, Scala, Kotlin, TypeScript, and AI frameworks into a single cohesive language. It provides a universal type system, tracked side effects, linear and affine memory management, first-class GPU and AI primitives, formal verification support, and 24 built-in domain engines covering everything from graph algorithms to quantum simulation.

This manual is the authoritative reference for the Chronos language and its compiler toolchain.

== Installation

=== Prerequisites

[cols="1,2,1", options="header"]
|===
|Dependency |Purpose |Required

|Rust toolchain (stable)
|Building the compiler itself
|Yes

|MSVC Build Tools _or_ GCC / Clang

|Yes

|Native Windows compilation via `cl.exe`
|MSVC only

On Windows the compiler automatically locates `cl.exe` in the standard Visual Studio installation paths.
On Linux and macOS it invokes `cc` (GCC or Clang).

=== Building the Compiler

[source,shell]
----
git clone https://github.com/chronos-lang/chronos
cd chronos
cargo build --release -p chronos-cli
----

The resulting binary is placed at `target/release/chronos` (or `chronos.exe` on Windows).
Add it to your `PATH` to use the `chronos` command globally.

=== Verifying the Installation

[source,shell]
----
chronos version
# Chronos 0.1.0
----

[[quick-start]]
== Quick Start

=== Your First Program

Create a file named `hello.chr`:

[source,chronos]
----
fn main() {
    let message: string = "Hello, Chronos!";
    println(message);
}
----

Compile and run it:

[source,shell]
----
chronos run hello.chr
# Hello, Chronos!
----

Or compile to a binary and run separately:

[source,shell]
----
chronos build hello.chr
./hello
----

=== A More Complete Example

[source,chronos]
----
// fibonacci.chr — demonstrates functions, loops, and mutable variables

fn fib_recursive(n: i64) -> i64 {
    if n <= 1 {
        return n;
    }
    return fib_recursive(n - 1) + fib_recursive(n - 2);
}

fn fib_iterative(n: i64) -> i64 {
    if n <= 1 {
        return n;
    }
    var a: i64 = 0;
    var b: i64 = 1;
    var i: i64 = 2;
    while i <= n {
        let tmp: i64 = a + b;
        a = b;
        b = tmp;
        i = i + 1;
    }
    return b;
}

fn main() {
    var idx: i64 = 0;
    while idx <= 10 {
        let r:  i64 = fib_recursive(idx);
        let it: i64 = fib_iterative(idx);
        println(idx, r, it);
        idx = idx + 1;
    }
}
----

[source,shell]
----
chronos run fibonacci.chr
# 0  0  0
# 1  1  1
# 2  1  1
# ...
# 10 55 55
----

[[part-language]]
= Part II: Language Reference

[[lexical-structure]]
== Lexical Structure

=== Source Encoding

Chronos source files are UTF-8 encoded. The conventional file extension is `.chr`.

=== Comments

[source,chronos]
----
// Single-line comment — extends to end of line

/* Block comment — can span
   multiple lines */
----

Block comments do not nest.

=== Whitespace

Horizontal whitespace (spaces, tabs) is ignored except as a separator between tokens.
Newlines are significant only in limited contexts (e.g., implicit semicolon insertion is not performed; all statements require an explicit terminator or closing brace).

=== Identifiers

An identifier starts with a letter (`a–z`, `A–Z`) or an underscore (`_`), followed by any combination of letters, digits (`0–9`), and underscores.

[source,chronos]
----
my_variable
_private
CamelCase
x1
----

=== Literals

==== Integer Literals

[cols="1,2,2", options="header"]
|===
|Form |Example |Description

|Decimal
|`42`, `1_000_000`
|Digits `0–9`, underscore separators allowed

|Hexadecimal
|`0xFF`, `0x1A_2B`
|Prefix `0x` or `0X`

|Octal
|`0o77`, `0o755`
|Prefix `0o` or `0O`

|Binary
|`0b1010`, `0b1111_0000`
|Prefix `0b` or `0B`
|===

The type of an integer literal is inferred from context; it defaults to `i64`.

==== Float Literals

[source,chronos]
----
3.14
1.0e10
1e-3
0.5
----

The type defaults to `f64`.

==== String Literals

[cols="1,2,2", options="header"]
|===
|Form |Example |Description

|Standard
|`"hello\nworld"`
|Double-quoted; supports escape sequences

|Raw
|`r"no \escapes"`
|`r"..."` prefix; backslashes are literal

|Multi-line
|`"""line one`
`line two"""`
|Triple-quoted; preserves newlines
|===

Common escape sequences: `\n`, `\r`, `\t`, `\\`, `\"`, `\xNN`, `\uNNNN`.

==== Character Literals

[source,chronos]
----
'a'
'\n'
'\x41'   // 'A'
----

==== Boolean Literals

`true` and `false`.

[[types]]
== Type System

Chronos has a rich static type system that combines ideas from Haskell (Hindley-Milner inference), Rust (ownership, lifetimes), Scala/Kotlin (data classes, sealed hierarchies), and dependent type theory (proof engine integration).

=== Primitive Types

==== Integer Types

[cols="1,2,2", options="header"]
|===
|Type |Width |Range

|`i8`  |8-bit signed   |−128 to 127
|`i16` |16-bit signed  |−32,768 to 32,767
|`i32` |32-bit signed  |−2^31 to 2^31−1
|`i64` |64-bit signed  |−2^63 to 2^63−1
|`i128`|128-bit signed |−2^127 to 2^127−1
|`int` |Arbitrary precision |Unbounded
|`u8`  |8-bit unsigned  |0 to 255
|`u16` |16-bit unsigned |0 to 65,535
|`u32` |32-bit unsigned |0 to 2^32−1
|`u64` |64-bit unsigned |0 to 2^64−1
|`u128`|128-bit unsigned|0 to 2^128−1
|`uint`|Arbitrary precision|≥ 0
|`usize`|Pointer-sized unsigned|Platform-dependent
|`isize`|Pointer-sized signed|Platform-dependent
|===

==== Float Types

[cols="1,2,2", options="header"]
|===
|Type |Width |Notes

|`f16`  |16-bit IEEE 754 half precision   |
|`f32`  |32-bit IEEE 754 single precision |
|`f64`  |64-bit IEEE 754 double precision |Default float type
|`f128` |128-bit IEEE 754 quad precision  |
|`bf16` |16-bit Brain Float               |For ML/AI workloads
|===

==== Other Primitives

[cols="1,3", options="header"]
|===
|Type |Description

|`bool`   |Boolean: `true` or `false`
|`char`   |Unicode scalar value (32-bit)
|`str`    |Borrowed string slice (UTF-8)
|`string` |Owned heap-allocated UTF-8 string
|`void`   |The unit/empty type; used for functions with no return value
|`never`  |The bottom type; the type of expressions that never produce a value (diverge, panic, infinite loop)
|===

=== Compound Types

==== Tuples

[source,chronos]
----
let pair: (i32, string) = (42, "hello");
let triple: (f64, f64, f64) = (1.0, 2.0, 3.0);
----

==== Arrays

Fixed-length, stack-allocated sequences:

[source,chronos]
----
let arr: [i32; 5] = [1, 2, 3, 4, 5];
----

==== Slices

Dynamically-sized views into a sequence:

[source,chronos]
----
let s: [i32] = arr[1..4];
----

==== Optional / Nullable

[source,chronos]
----
let maybe: i32? = some(42);
let nothing: i32? = none;
----

The postfix `?` syntax is shorthand for `Optional<T>`.

=== Function Types

[source,chronos]
----
let f: fn(i32, i32) -> i32 = |a, b| a + b;
----

Function types may carry an effect signature:

[source,chronos]
----
let g: fn(string) -> void / IO;
----

=== Memory Ownership Types

Chronos provides three ownership qualifiers that eliminate garbage collection:

[cols="1,2,2", options="header"]
|===
|Qualifier |Semantics |C++ analogy

|`T` (default)
|Owned value; moved or copied
|Value type / `unique_ptr`

|`linear T`
|Must be consumed exactly once
|Move-only with mandatory use

|`affine T`
|May be consumed at most once
|`unique_ptr` (optional drop)
|===

[source,chronos]
----
let buf: linear [u8; 1024] = allocate_buffer();
process(buf);   // buf is consumed here; cannot use again

let handle: affine File = open("log.txt");
// handle may be dropped without explicit consumption
----

=== References

[source,chronos]
----
let x: i32 = 10;
let r: &i32 = &x;          // immutable borrow
let rw: &mut i32 = &mut x; // mutable borrow
----

=== Tensor Types

First-class tensors for GPU/ML code:

[source,chronos]
----
let weights: tensor<f32, [768, 3072]>;
let image:   tensor<f32, [N, C, H, W]>;   // symbolic shape
let output:  tensor<f32, [_]>;             // dynamic shape
----

A tensor type carries its element type, shape, and target device.

=== Generic Types

[source,chronos]
----
struct Pair<A, B> {
    first:  A,
    second: B,
}

fn identity<T>(x: T) -> T {
    return x;
}

fn show<T: Display>(value: T) where T: Clone {
    println(value.to_string());
}
----

=== Type Aliases

[source,chronos]
----
type Meters = f64;
type Grid<T> = Vec<Vec<T>>;
type Callback = fn(i32) -> void;
----

=== Effects

Every function type may carry an effect set — a compile-tracked annotation of observable side effects:

[cols="1,3", options="header"]
|===
|Effect |Meaning

|`IO`       |Reads or writes I/O (files, stdin/stdout, etc.)
|`Alloc`    |Performs heap allocation
|`Async`    |Suspends execution (coroutine / future)
|`Gpu`      |Executes on a GPU
|`Tpu`      |Executes on a TPU
|`Npu`      |Executes on an NPU
|`Network`  |Accesses the network
|`Throw(E)` |May throw an exception of type `E`
|`Diverge`  |May not terminate
|`Quantum`  |Uses quantum operations
|`Pure`     |No observable side effects
|`Custom(s)`|User-defined effect label
|===

[source,chronos]
----
pure fn add(a: i32, b: i32) -> i32 {
    return a + b;
}

async fn fetch(url: string) -> string / IO, Network, Alloc {
    // ...
}
----

[[variables]]
== Variables and Bindings

=== Immutable Bindings (`let`)

[source,chronos]
----
let x: i32 = 42;
let name: string = "Alice";
let computed = some_function();   // type inferred
----

`let` bindings are immutable by default. The type annotation is optional when it can be inferred.

=== Mutable Bindings (`var`)

[source,chronos]
----
var counter: i64 = 0;
counter = counter + 1;
counter += 1;    // compound assignment
----

`var` declares a mutable variable. All mutation requires the `var` keyword at the declaration site.

=== `val` — Immutable Binding (Scala/Kotlin style)

`val` is an alias for `let`.

=== `const` — Compile-Time Constants

[source,chronos]
----
const MAX_SIZE: usize = 1024;
const PI: f64 = 3.14159265358979;
----

`const` values are evaluated at compile time and inlined at every use site.

=== Assignment and Compound Assignment

[cols="1,2", options="header"]
|===
|Operator |Meaning

|`x = expr`  |Assign
|`x += expr` |Add and assign
|`x -= expr` |Subtract and assign
|`x *= expr` |Multiply and assign
|`x /= expr` |Divide and assign
|`x %= expr` |Modulo and assign
|===

[[functions]]
== Functions

=== Basic Declaration

[source,chronos]
----
fn add(a: i32, b: i32) -> i32 {
    return a + b;
}

fn greet(name: string) {
    println("Hello, ", name);
}
----

The return type follows `->`. It may be omitted when the function returns `void`.
Both `fn` and `fun` are accepted as the function keyword.

=== Expression-Body Shorthand

[source,chronos]
----
fn square(x: i32) -> i32 = x * x;
----

=== Visibility Modifiers

[source,chronos]
----
pub fn public_api() { }
private fn internal_helper() { }
protected fn subclass_api() { }
internal fn module_api() { }
crate fn crate_api() { }
----

=== Async Functions

[source,chronos]
----
async fn fetch_data(url: string) -> string {
    let response = await http_get(url);
    return response.body;
}
----

=== Inline and Constexpr

[source,chronos]
----
inline fn hot_path(x: i32) -> i32 { return x * 2; }
constexpr fn factorial(n: u32) -> u32 { /* ... */ }
----

=== Operator Overloading

[source,chronos]
----
operator fn +(self: Vec2, other: Vec2) -> Vec2 {
    return Vec2 { x: self.x + other.x, y: self.y + other.y };
}
----

=== Degradable Functions

Functions annotated with `degradable` carry an expiry date. The compiler emits warnings as the expiry approaches and errors once it passes.

[source,chronos]
----
@expires("2027-01-01")
@warns("2026-06-01")
@replaces("new_compute")
degradable fn old_compute(x: f32) -> f32 {
    // ...
}
----

[[control-flow]]
== Control Flow

=== `if` / `else if` / `else`

[source,chronos]
----
if score >= 90 {
    println("A");
} else if score >= 80 {
    println("B");
} else {
    println("C");
}
----

Conditions do not require parentheses. `if` is also an expression:

[source,chronos]
----
let label: string = if x > 0 { "positive" } else { "non-positive" };
----

=== `while`

[source,chronos]
----
var i: i64 = 0;
while i < 10 {
    println(i);
    i += 1;
}
----

=== `for` / `in`

[source,chronos]
----
for item in collection {
    println(item);
}

for i in 0..10 {    // range: exclusive upper bound
    println(i);
}

for i in 0..=10 {   // range: inclusive upper bound
    println(i);
}
----

=== `loop`

An infinite loop that must be exited with `break` or `return`:

[source,chronos]
----
loop {
    let input = read_line();
    if input == "quit" {
        break;
    }
}
----

=== `break` and `continue`

[source,chronos]
----
while condition {
    if skip_this { continue; }
    if stop_now  { break; }
    process();
}
----

=== `return`

[source,chronos]
----
fn find(haystack: [i32], needle: i32) -> i32? {
    for item in haystack {
        if item == needle { return some(item); }
    }
    return none;
}
----

=== `match` / `when`

[source,chronos]
----
match direction {
    Direction::North => move_north(),
    Direction::South => move_south(),
    Direction::East | Direction::West => move_horizontal(),
    _ => println("unknown"),
}
----

`when` is an accepted alias for `match` (Kotlin style).

Guards narrow a pattern further:

[source,chronos]
----
match value {
    x if x < 0  => println("negative"),
    0            => println("zero"),
    x if x > 100 => println("large"),
    x            => println(x),
}
----

=== `try` / `catch` / `finally`

[source,chronos]
----
try {
    let result = risky_operation();
    return result;
} catch (e: IOException) {
    println("I/O error: ", e.message);
} finally {
    cleanup();
}
----

=== `defer`

Schedules a statement to run when the enclosing scope exits:

[source,chronos]
----
fn read_file(path: string) -> string {
    let f = open(path);
    defer close(f);        // runs on all exit paths
    return f.read_all();
}
----

=== `drop`

Explicitly destroys a linear or affine resource before its scope ends:

[source,chronos]
----
{
    let buf: linear [u8; 4096] = alloc();
    // ... use buf ...
    drop buf;   // mandatory for linear types
}
----

[[structs-classes]]
== Structs and Classes

=== Struct Declarations

[source,chronos]
----
pub struct Point {
    x: f64,
    y: f64,
}

pub struct Rect {
    origin: Point,
    width:  f64,
    height: f64,
}
----

=== Struct Literals

[source,chronos]
----
let p = Point { x: 3.0, y: 4.0 };

let r = Rect {
    origin: p,
    width:  10.0,
    height: 5.0,
};
----

=== `impl` Blocks

[source,chronos]
----
impl Point {
    fn new(x: f64, y: f64) -> Point {
        return Point { x: x, y: y };
    }

    fn distance(self: Point, other: Point) -> f64 {
        let dx = self.x - other.x;
        let dy = self.y - other.y;
        return dx * dx + dy * dy;
    }

    fn translate(self: Point, dx: f64, dy: f64) -> Point {
        return Point { x: self.x + dx, y: self.y + dy };
    }
}
----

Methods take an explicit `self` parameter of the implementing type.
Static/associated functions omit `self`.

=== Class Declarations

[source,chronos]
----
class Animal {
    let name: string;
    var energy: i32;

    fn new(name: string) -> Animal {
        return Animal { name: name, energy: 100 };
    }

    fn speak(self: Animal) {
        println(self.name, "says something.");
    }
}

class Dog : Animal {
    override fn speak(self: Dog) {
        println(self.name, "barks!");
    }
}

abstract class Shape {
    abstract fn area(self: Shape) -> f64;
}

final class Circle : Shape {
    let radius: f64;

    override fn area(self: Circle) -> f64 {
        return 3.14159 * self.radius * self.radius;
    }
}
----

Class modifiers: `abstract`, `final`, `open`, `sealed`.

=== Data Classes

Kotlin-style data classes auto-derive equality, hashing, and a `copy()` method:

[source,chronos]
----
data class Point(x: f64, y: f64)
data class Person(name: string, age: i32)
----

=== Companion Objects

[source,chronos]
----
class Counter {
    var count: i32;

    companion object {
        let DEFAULT: i32 = 0;

        fn create() -> Counter {
            return Counter { count: Counter::DEFAULT };
        }
    }
}
----

=== Field Access

[source,chronos]
----
let area = rect.width * rect.height;
let x_coord = rect.origin.x;
----

=== Safe Navigation

[source,chronos]
----
let length = maybe_string?.len();
----

Returns `none` if `maybe_string` is `none`; otherwise applies `.len()`.

[[traits]]
== Traits and Interfaces

=== Declaring a Trait

[source,chronos]
----
trait Display {
    fn fmt(self: Self) -> string;
}

trait Printable : Display + Clone {
    fn print(self: Self) {
        println(self.fmt());      // default implementation
    }
}
----

`interface` is accepted as a synonym for `trait`.

=== Implementing a Trait

[source,chronos]
----
impl Display for Point {
    fn fmt(self: Point) -> string {
        return "(" + self.x.to_string() + ", " + self.y.to_string() + ")";
    }
}
----

=== Generic Bounds

[source,chronos]
----
fn print_all<T: Display>(items: Vec<T>) {
    for item in items {
        println(item.fmt());
    }
}

fn sort<T>(items: Vec<T>) where T: Ord + Clone { /* ... */ }
----

=== `derive`

Auto-generate common trait implementations:

[source,chronos]
----
@derive(Equals, Hash, ToString, Debug)
struct Color {
    r: u8,
    g: u8,
    b: u8,
}
----

Available derivable traits: `Equals`, `Hash`, `ToString`, `Copy`, `Clone`, `Debug`, `Serialize`, `Order`, `Destructure`.

== Generics and Templates

=== Generic Functions

[source,chronos]
----
fn max<T: Ord>(a: T, b: T) -> T {
    if a > b { return a; } else { return b; }
}
----

=== Generic Structs

[source,chronos]
----
struct Stack<T> {
    items: Vec<T>,
}

impl<T> Stack<T> {
    fn push(self: Stack<T>, item: T) -> Stack<T> { /* ... */ }
    fn pop(self: Stack<T>) -> (T?, Stack<T>) { /* ... */ }
}

[source,chronos]

template<typename T, usize N>

[source,chronos]
----
class Producer<out T> { fn produce(self: Self) -> T; }

`out` = covariant, `in` = contravariant.

[[operators]]
== Operators

=== Operator Precedence

Operators are listed from lowest precedence (binds loosest) to highest (binds tightest).
Left–right associativity means the left operand binds tighter when both sides have the same nominal precedence.

[cols="1,1,1,3", options="header"]
|===
|Operators |Left BP |Right BP |Description

|`=` `+=` `-=` `*=` `/=` `%=`
|2 |1
|Right-associative assignment

|`\|>`
|3 |4
|Pipe-forward (left)

|`\|\|`
|5 |6
|Logical OR

|`&&`
|7 |8
|Logical AND

|`==` `!=` `<` `>` `<=` `>=` `<=>`
|9 |10
|Comparison / spaceship

|`\|`
|11 |12
|Bitwise OR

|`^`
|13 |14
|Bitwise XOR

|`&`
|15 |16
|Bitwise AND

|`<<` `>>`
|17 |18
|Shift

|`..` `..=`
|19 |20
|Range (exclusive / inclusive)

|`+` `-`
|21 |22
|Addition, subtraction

|`*` `/` `%` `@`
|23 |24
|Multiply, divide, modulo, matrix multiply

|`**`
|28 |27
|Exponentiation (right-associative)

|`.` `?.` `::`
|29 |30
|Member access, safe navigation, path
|===

[cols="1,1,2", options="header"]
|===
|Prefix operator |Binding power |Description

|`-` `!` `~` `&` |25 |Unary minus, logical not, bitwise not, borrow
|`*` |25 |Dereference
|===

[cols="1,1,2", options="header"]
|===
|Postfix operator |Binding power |Description

|`?` |27 |Error propagation
|`!!`|27 |Non-null assertion
|`()`|27 |Function call
|`[]`|27 |Index
|===

=== Special Operators

[cols="1,3", options="header"]
|===
|Operator |Description

|`\|>`
|Pipe-forward: `x \|> f` is equivalent to `f(x)`

|`@`
|Matrix multiply (for tensors/matrices)

|`<=>`
|Three-way comparison (spaceship); returns negative/zero/positive

|`?.`
|Safe navigation: short-circuits to `none` if the left side is `none`

|`!!`
|Non-null assertion: panics at runtime if the value is `none`

|`..`
|Exclusive range: `1..5` produces `[1, 2, 3, 4]`

|`..=`
|Inclusive range: `1..=5` produces `[1, 2, 3, 4, 5]`
|===

[[memory-model]]
== Memory Model

=== Ownership

Every value has exactly one owner at any point in time. When the owner goes out of scope, the value is dropped.

[source,chronos]
----
{
    let s = String::from("hello");  // s owns the string
    use(s);                          // ownership transferred to use()
    // s is no longer valid here
}
----

=== Borrowing

[source,chronos]
----
fn print_len(s: &string) {
    println(s.len());
}

let msg = "hello";
print_len(&msg);    // borrow; msg is still valid after
----

Rules:
* Any number of shared (`&`) borrows may coexist.
* A mutable (`&mut`) borrow may not coexist with any other borrow.
* No borrow may outlive its referent.

=== Linear Types

A value of type `linear T` must be used exactly once. The compiler guarantees this at compile time.

[source,chronos]
----
fn encrypt(key: linear CryptoKey, data: [u8]) -> ([u8], linear CryptoKey) {
    // must return the key; cannot silently drop it
}
----

=== Affine Types

A value of type `affine T` may be used at most once. Unlike linear, it may be silently discarded.

[source,chronos]
----
let handle: affine FileHandle = open("log.txt");
// handle may be dropped by going out of scope without explicit consumption
----

=== `with` — Resource Scoping

[source,chronos]
----

    let content = f.read_all();
}   // f is automatically closed here
----

=== Compilation Strategy

The compiler uses a two-pass strategy:

1. **C back-end (primary)** — emits C99 code and compiles with:
** Windows: `cl.exe` (MSVC) located automatically in standard VS installation paths
** Linux / macOS: `cc` (GCC or Clang)
2. **LLVM IR back-end (fallback)** — emits LLVM IR and invokes `clang` when the C path is unavailable or `--emit-llvm` is specified.

When a compilation error occurs from the underlying C compiler, the generated `.c` file is preserved and a `Compile manually:` hint is printed so you can inspect it directly.

[[compiler-pipeline]]
== Compiler Pipeline

Every source file passes through eight sequential phases. All phases share a `CompilationContext` structure that accumulates results.

[plantuml, pipeline, svg]
....
Source → Lexer → Parser → Type Inference → Security Audit → Degradation Check → IR Lowering → WCET Analysis → Code Generation
....

=== Phase 1 — Lexer

*Input:* UTF-8 source text +
*Output:* `Vec<SpannedToken>` +
*Module:* `chronos-lexer.rs`

The lexer is generated from the `logos` crate via a DFA. It recognizes:

* 150+ keyword tokens (organized by feature area)
* All numeric literal forms (decimal, hex, octal, binary)
* Standard, raw, and triple-quoted string literals
* Unicode identifiers
* All operator and punctuation tokens
* Line comments, block comments, and whitespace (all automatically skipped)

The output retains source span (line/column) information for every token, used in diagnostic messages.

=== Phase 2 — Parser

*Input:* Token stream +
*Output:* `Program` AST +
*Module:* `chronos-parser.rs`, `chronos-parser-v2.rs`

The parser is a hand-written recursive descent / Pratt parser. The Pratt algorithm handles operator precedence in expressions without grammar ambiguity.

Recognized top-level items:

`fn`, `async fn`, `kernel`, `class`, `abstract class`, `struct`, `data class`, `enum`, `sealed enum`, `trait`, `interface`, `impl`, `impl … for`, `type`, `mod`, `import`, `use`, `const`, `static`, `ai skill`, `ai tool`, `ai pipeline`, `degradable fn`

=== Phase 3 — Type Inference

*Input:* `Program` AST +
*Output:* `TypeEnvironment`, inferred effects, taint map, unit map +
*Module:* `chronos-type-inference.rs`, `chronos-inference-ir-v2.rs`

The type inference engine implements Hindley-Milner extended with:

* **Subtyping** — numeric coercions, trait bounds
* **Linear / affine type checking** — consume-exactly-once semantics
* **Effect inference** — tracks IO, Alloc, Gpu, Network, etc. per function
* **Taint tracking** — data-flow analysis that records which variables hold potentially unsafe external data
* **Dimensional analysis** — detects unit mismatches (e.g., metres + seconds)
* **Quantum state tracking** — models qubit state through operations

A forward-declaration pre-pass registers all function signatures before their bodies are checked, enabling mutual recursion and out-of-order definitions.

=== Phase 4 — Security Audit

*Input:* `CompilationContext` (especially taint map) +
*Output:* `Vec<SecurityFinding>`, `audit_passed: bool` +
*Module:* `chronos-security-realtime.rs`

Activated by `#![secure]` at the top of the source file or the `--secure` flag.
The audit engine runs multiple passes:

* Buffer overflow / bounds checking
* Injection vulnerabilities (SQL, command, XSS, path traversal)
* Cryptographic weaknesses (weak ciphers, weak hash functions, hardcoded credentials)
* Memory safety (use-after-free, double-free)
* Access control and authentication gaps
* Race conditions

Each finding is tagged with a severity, a CWE identifier, a CVSS score, and an auto-fix suggestion where applicable.

==== Security Modes

[cols="1,3", options="header"]
|===
|Mode |Behavior

|`Off` (default)
|No audit; fastest build

|`Warn`
|Audit runs; findings printed as warnings; compilation succeeds

|`Enforce`
|Any `Medium` or higher finding causes a compilation error

|`Formal`
|`Enforce` + formal verification + mandatory code signing
|===

Activate via source annotation:

[source,chronos]
----
#![secure(enforce)]

fn handle_user_input(raw: string) {
    // taint-tracked; unsafe use causes compile error
}
----

=== Phase 5 — Degradation Check

*Input:* AST (degradable function annotations) +
*Output:* `Vec<CompilerDiagnostic>` +
*Module:* `chronos-unified-integration.rs`

Checks `@expires` and `@warns` dates against the current date and emits warnings or errors accordingly. This enforces automatic deprecation timelines.

=== Phase 6 — IR Lowering

*Input:* `Program` AST, `TypeEnvironment` +
*Output:* `IRModule` (SSA-based IR) +
*Module:* `chronos-ir-codegen.rs`

The IR is an SSA (Static Single Assignment) form inspired by MLIR. It supports:

* Standard arithmetic, comparison, and memory opcodes
* High-level domain-specific opcodes: `Gemm` (matrix multiply), `Conv`, `Scatter`, `Gather`
* GPU-specific opcodes for CUDA/NVPTX target
* Quantum opcodes for the quantum simulation target

Three code generation back-ends read from the IR:

* `LLVMBackend` — emits LLVM IR text (`.ll`)
* `CUDABackend` — emits NVPTX PTX assembly
* `CBackend` — emits C99 source (`.c`), the primary production back-end

=== Phase 7 — WCET Analysis (Real-Time Mode)

*Input:* `IRModule` +
*Output:* `HashMap<fn_name, WCETResult>` +
*Module:* `chronos-security-realtime.rs`

Activated by `#![realtime]` or `--realtime`. Performs static worst-case execution time analysis, computing cycle-count upper bounds for each function relative to a target `HardwareModel`.

Real-time mode also enforces no dynamic allocation (`Alloc` effect forbidden) and no unbounded loops.

[source,chronos]
----
#![realtime]

fn isr_handler(event: u32) {
    // must complete within a bounded cycle count
}
----

=== Phase 8 — Code Generation

*Input:* `IRModule`, AST +
*Output:* Byte blobs in `CompilationContext::generated_code` +
*Module:* `chronos-c-backend.rs`, `chronos-ir-codegen.rs`

==== C Back-End (Primary)

The C back-end lowers the Chronos AST directly to C99 without going through IR, providing maximal compatibility with MSVC, GCC, and Clang. It:

* Emits typed C declarations (`int64_t`, `double`, `const char*`, etc.)
* Emits a small runtime prelude with helpers like `__chr_println_str`, `__chr_println_i64`, and C11 `_Generic` macros for type-dispatched printing
* Handles `if`/`else`, `while`, assignments, function calls, struct literals, and field access
* Emits forward declarations for all functions to permit mutual recursion

Generated C is compiled by the platform C compiler:

[cols="1,2", options="header"]
|===
|Platform |Compiler

|Windows (MSVC)
|`cl.exe /std:c11 /O2`

|Linux / macOS
|`cc -std=c11 -O2 -lm`
|===

==== LLVM IR Back-End (Fallback)

Emits LLVM IR text suitable for `llc` or `clang`. Activated by `--emit-llvm` or when `cl.exe` and `cc` are both unavailable.

[[compilation-context]]
== CompilationContext — Shared Pipeline State

All eight compiler phases read and write a single `CompilationContext` structure. This eliminates the need for inter-phase function arguments and makes the pipeline trivially extensible.

[source,rust]
----
pub struct CompilationContext {
    pub filename:             String,
    pub source:               String,
    pub security_mode:        SecurityMode,
    pub realtime_mode:        bool,

    // Phase 1 outputs
    pub tokens:               Vec<SpannedToken>,

    // Phase 2 outputs
    pub ast:                  Option<Program>,
    pub version_rules:        Option<VersionRules>,

    // Phase 3 outputs
    pub type_env:             TypeEnvironment,
    pub inferred_effects:     HashMap<String, Vec<Effect>>,
    pub taint_map:            TaintMap,
    pub unit_map:             UnitMap,
    pub qubit_state_map:      QubitStateMap,
    pub linear_resource_map:  LinearResourceMap,
    pub proof_obligations:    Vec<ProofObligation>,

    // Phase 4 outputs
    pub security_findings:    Vec<SecurityFinding>,
    pub audit_passed:         bool,

    // Phase 5 outputs
    pub degradation_diagnostics: Vec<CompilerDiagnostic>,

    // Phase 6 outputs
    pub ir_module:            Option<IRModule>,

    // Phase 7 outputs
    pub wcet_results:         HashMap<String, WCETResult>,

    // Phase 8 outputs
    pub generated_code:       HashMap<String, Vec<u8>>,

    pub all_diagnostics:      Vec<CompilerDiagnostic>,
}
----

The `generated_code` map is keyed by back-end name: `"c"` for the C back-end, `"llvm"` for LLVM IR, `"ptx"` for CUDA PTX.

[[include-order]]
== compiler-core Include Order

The `chronos-compiler-core` crate includes all pipeline files sequentially in a single Rust module via relative `include!()` paths under `src/compiler/`, so every type defined in an earlier file is in scope for all later files:

[cols="1,3", options="header"]
|===
|Step |File (relative to workspace root)

|1  |`src/compiler/chronos-lexer.rs` — defines `Token`, `Lexer`, `SpannedToken`
|2  |`src/compiler/chronos-stdlib-types.rs` — universal `ChronosType` enum
|3  |`missing_types.rs` — AST nodes: `Expression`, `Statement`, `Program`, `BinOp`, … _(in crate src/)_
|4  |`src/compiler/chronos-parser.rs` — recursive descent / Pratt parser
|5  |`src/compiler/chronos-parser-v2.rs` — extended parser for domain-specific syntax
|6  |`src/compiler/chronos-type-inference.rs` — Hindley-Milner + effects + taint
|7  |`src/compiler/chronos-inference-ir-v2.rs` — V2 type rules
|8  |`src/compiler/chronos-ir-codegen.rs` — SSA IR + LLVM / CUDA back-ends
|9  |`src/compiler/chronos-security-realtime.rs` — 9-pass audit + WCET analysis
|10 |`src/compiler/chronos-unified-integration.rs` — `compile()` orchestrator
|11 |`src/compiler/chronos-c-backend.rs` — C99 code generator
|12 |`missing_types_late.rs` — wrapper types delegating to real implementations _(in crate src/)_
|===

[[appendix-diagnostics]]
[appendix]
== Diagnostic Codes

Error codes follow the pattern `E-PHASE-NNN`.

[cols="1,1,3", options="header"]
|===
|Code prefix |Phase |Description

|`E-LEX-`    |Lexer          |Lexical error (invalid character, unterminated literal)
|`E-PARSE-`  |Parser         |Syntax error (unexpected token, missing brace)
|`E-TYPE-`   |Type inference |Type mismatch, unbound variable, arity mismatch
|`E-SEC-`    |Security audit |Vulnerability detected
|`E-DEGRAD-` |Degradation    |Expired or imminently expiring function
|`E-IR-`     |IR lowering    |Lowering error
|`E-WCET-`   |WCET analysis  |Real-time constraint violated
|`E-CODEGEN-`|Code generation|Code generation failure
|===

Common type errors:

[cols="1,3", options="header"]
|===
|Code |Meaning

|`E-TYPE-001` |Unbound variable: identifier not in scope
|`E-TYPE-002` |Type mismatch: expected X, found Y
|`E-TYPE-003` |Function arity mismatch: wrong number of arguments
|`E-TYPE-004` |Linear resource not consumed
|`E-TYPE-005` |Linear resource used more than once
|===
